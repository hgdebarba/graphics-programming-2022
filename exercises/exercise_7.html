Initial implementation

For the exercise this week we will continue with the car scene from exercise 5.

The code has been improved in a few ways:

  • The car model has been loading in pieces, allowing to modify only one material (the car paint).
  • Instead of 2 lights in the shader, we have implemented a forward rendering model, with a loop to render additional passes for other lights. The shader is a version of phong_shading in exercise 5.

On top of that we have prepared the skeleton of a deferred rendering model for you to complete. The g-buffers used in this exercise are:

  • Albedo: The reflection color.
  • Normal: Normal of the objects in view space.
  • Others: Reflectance constants and specular exponent.
  • Depth: We will use the depth buffer to reconstruct the position of the fragments.

The deferred renderer also renders in additive passes, but each pass only needs to cover the pixels for that specific light. For that, we render cubes with the size of the light radius. Only the first pass is a full screen quad, to make sure that we apply ambient to all the pixels.

You can use the GUI to swap between the two rendering modes: forward and deferred.

Take some time to get familiar with the C++ code and observe the differences between forward and deferred paths.
Exercise 7.1
 Light radius attenuation

In exercise 5 we implemented attenuation based on distance. While this is physically accurate, sometimes it is preferred to limit the radius of a light for performance reasons.

For simplicity, we will use GLSL function smoothstep. This function takes two values as a range (min, max) and a third value to sample in that range. At min, the function returns 1. At max, the function returns 0. In between, it smoothly interpolates from 0 to 1 depending on the position in that range. For us, some good values are:
smoothstep(lightRadius, lightRadius * 0.5f, distanceToLight)

The lights in this exercise already have a setting to control the radius, which you can change using the UI. But currently it does not have any effect, it needs to be implemented in our shaders:

  • Find where we compute the attenuation in forward_shading.frag.
  • Compute the distance falloff using the function above.
  • Multiply the current attenuation by that falloff.

Try changing the radius in the UI and observe the behavior.

Exercise 7.3
Deferred rendering - Albedo

The first g-buffer that we are going to work with is the Albedo. This g-buffer contains the color of the objects in the scene.

As just passing the reflection color would be too easy, why not add some textures here? The exercise is already sampling the right textures in C++ code, you just need to handle the shader part:

  • In deferred_shading.vert, get the vertex texture coordinates from the declared attribute and pass it to the fragment shader
  • In deferred_shading.frag, receive the texture coordinates from the vertex shader.
  • In the same file, declare a sampler uniform and use the texture coordinates to sample from it.
  • We still want to be able to change the color of the car paint, so we multiply the sampled albedo with the reflection color.
  • Finally, in lighting.frag, comment out the debug output and uncomment the albedo output. Observe the content of the albedo g-buffer when selecting Deferred shading!

To be able to compare with forward, repeat the first 4 steps in forward_shading.vert and forward_shading.frag.
Exercise 7.4
Deferred rendering - Normal

To be able to compute the lighting, we need to store the normal of the objects. We mentioned in class that we will be computing the lighting in view space, so all the vectors and positions involved need to be in that space, including the normal.

Remember: to transform from model space to view space, first we multiply by the model matrix, and then by the view matrix. Also, if you want to do the operations inline, remember that the transformations are applied from right to left

  • In deferred_shading.vert, compute the normal in view space and pass it to the fragment shader.
  • In deferred_shading.frag, get the normal from the vertex shader, normalize it, and output the value to NormalGBuffer. Notice how only 2 components are being stored. The third one is implicit.
  • in lighting.frag we are already reading the value from the normal. However, we still need to reconstruct that third component! To do so, we apply the formula we have seen in class, knowing that the length must be one.
  • Comment out the FragColor from the previous exercise and uncomment the normal output. Look at the colors that it produces. Can you see how it changes the color when you move the camera? That is because it is in view space.

One thing worth mentioning here is that the format used for this g-buffer (RG16F) can take negative values, that is why we don't need to transform it to a [0, 1] range. Therefore the normal don't look blueish like in the normal maps.
Exercise 7.5
Deferred rendering - Others

Ok, so we have the position (from depth), the normal, the albedo, the light properties... what else do we need?

What about the ambient, diffuse, and specular reflectance? We need to store those as well. The range for those is already [0, 1], so we need to do much with those.

There is one last value that we need, the specular exponent. The maximum value that we are using for this is 100, can we use that information to transform to the range [0, 1]?.

  • In deferred_shading.frag, store these properties using one channel for each, making sure that their range is [0, 1].
  • In lighting.frag, unpack these properties into local variables. Remember to restore any transformations that you had to do to fit in the range.
  • As with the other g-buffers, comment out the previous FragColor output and uncomment the one for these g-buffer. Try modifying the properties in the UI and see how the content changes.
Exercise 7.6
Deferred rendering - Phong reflection model

Now that we have unpacked all the information from our g-buffers we are ready to compute the phong reflection model lighting.

You can start by copying the lighting from forward_shading.frag, but you will notice that we are still missing some parts:

  • The light vector: We can compute it as usual, since we have reconstructed the position in view space. If you look at how the light uniforms are set from C++, you can see that the light position is passed in view space, so this operation is valid.
  • The view vector: In view space the camera position is at the origin. We can use that to simplify how we compute the view vector. Just remove the camera position from the equation and we are done.
  • Finally, we are ready to compute the lighting! Make sure to uncomment the correct FragColor for this step.

Congratulations! You have finished your deferred renderer. Now you can compare the results from forward shading and deferred shading.
